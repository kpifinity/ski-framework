# Security defaults — reference implementation

> Status: this reflects the v0.1.0-alpha reference implementation. For
> production deployments, treat this document as the minimum bar and add
> organization-specific controls on top.

## What is hardened by default

| Control | Default | Where enforced |
|---|---|---|
| TLS between operators and the SKI Model service | **enabled** | `docker-compose.yml` (`TLS_ENABLED=true`), `ski_model.server`, self-signed certs generated by `scripts/setup.sh` |
| API authentication | **enabled** (HMAC-style `X-API-Key` header) | `ski_model.server.require_api_key`; `SKI_API_KEY` is a required env var |
| Postgres passwords | **must be operator-supplied**; stack refuses to start without them | `docker-compose.yml` uses `${VAR:?msg}` defaults |
| Postgres password hashing | **SCRAM-SHA-256** | `POSTGRES_INITDB_ARGS` |
| Audit ledger append-only | **enforced at DB layer** via triggers | `src/ledger/append_only.sql` |
| Audit ledger read-only role for BI / reporting users | **provided** as `ski_audit_reader` | `src/ledger/append_only.sql` |
| Knowledge Graph signature verification | **mandatory**; service refuses to load an unsigned KG by default | `src/ski_model/kg_loader.py`, `tools/ski-model-deploy` |
| Determinism canary | **enabled** by default, 5-minute interval | `src/ski_model/canary.py` |
| Containers run as non-root | **yes** (UID 10001/10002) | `Dockerfile.ski-model`, `Dockerfile.sidecar` |
| Kafka authentication | SASL/SCRAM, **enabled** | `docker-compose.yml`, `monitoring/kafka_jaas.conf` |
| Outbound network calls during inference (default backend) | **none** | Ollama runs in the `ski-internal` Docker network |
| pgAdmin | **disabled by default** | docker-compose profile `pgadmin` required to enable |
| Default LLM backend | local (Ollama) | `SKI_INFERENCE_BACKEND=ollama` |
| Anthropic API backend | **labelled non-conformant**; logs a loud warning on init | `src/ski_model/backends.py::AnthropicDemoBackend` |

## What is deferred to the operator

These are intentionally out of scope for the alpha. Make decisions about
them before production.

| Concern | Notes |
|---|---|
| Secrets management integration (Vault, AWS Secrets Manager, GCP Secret Manager) | The reference implementation reads secrets from environment variables. Operators wire them up to their preferred manager. A sample HashiCorp Vault integration is sketched in [`docs/CUSTOMIZATION.md`](./docs/CUSTOMIZATION.md). |
| Certificate authority | The setup script generates self-signed certs for local use. For production, replace `tls/` with certs from your own CA. Rotate at the cadence your CA mandates. |
| Disk encryption at rest | Use the encryption facilities of the underlying volume / storage class. Not part of the docker-compose stack. |
| Backup encryption | `audit-ledger` produces a `pg_dump` and a SHA-256 checksum. Encrypting the backup at rest is the operator's responsibility (e.g. `age`, KMS-wrapped key). |
| Air-gapped operation | Set `networks.ski-internal.internal: true` in `docker-compose.yml` and pre-stage container images + Ollama model files on physical media. Documented in [`docs/AIR_GAPPED.md`](./docs/AIR_GAPPED.md). |
| Hardware baseline | The Determinism Enforcement Controls (B3.4) require pinning hardware so an organisation can audit reproducibility. Document your CPU model, instruction-set extensions, OS kernel, container runtime, and Ollama version against the manifest in [`docs/HARDWARE_BASELINE.md`](./docs/HARDWARE_BASELINE.md). |

## Outbound network calls

In the **default configuration**, the reference implementation makes no
outbound calls during inference. The only outbound traffic at runtime is:

- `ollama` ↔ `ski-model` (inside the `ski-internal` Docker network)
- `sidecar` → `ski-model` (inside the `ski-internal` Docker network)
- `ski-model` → `postgres` (inside the `ski-internal` Docker network)
- `prometheus` → exporters (inside the `ski-internal` Docker network)

If you enable the **anthropic-demo** backend, the stack will additionally
issue HTTPS calls to `api.anthropic.com`. The service logs a loud warning
on every startup and on every evaluation when this backend is active. Use
this mode only for local evaluation; it disqualifies the deployment from
all SKI conformance levels.

## Secrets bootstrap

The `scripts/setup.sh` helper:

1. Generates a 32-byte random `SKI_API_KEY` (hex-encoded).
2. Generates strong random Postgres and Grafana passwords.
3. Generates a self-signed CA and per-service TLS certs under `tls/`.
4. Writes everything to `reference-implementation/.env` with restrictive
   file permissions (0600).
5. Prints a one-time summary of where secrets live and refuses to print
   them to logs again.

Treat the generated `.env` file as a secret. Do not commit it. The
`.gitignore` already excludes it.

## What to do if you find a security defect

See [`SECURITY.md`](../SECURITY.md). Do not open a public issue.

## Metrics endpoint

`GET /metrics` (Prometheus exposition) is **unauthenticated by
design** — scrape agents cannot carry the API key. It exposes aggregate
operational counters only: verdict counts by taxonomy type, agreement
rate, KG-signature status, latency histograms. No measurement payloads,
verdict bodies, reasoning text, or tenant identifiers are ever exported.
Network exposure is bounded the same way as every other port: the
compose network / Helm NetworkPolicy keep it inside the sovereign
boundary. If your threat model requires it, front the port with an
authenticating scrape proxy.
